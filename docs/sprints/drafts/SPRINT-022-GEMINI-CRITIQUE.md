# Sprint 022: Draft Critique

## 1. Claude Draft Critique (`SPRINT-022-CLAUDE-DRAFT.md`)

### Strengths
*   **Mathematical Detail:** Provides an excellent, highly detailed breakdown of the OSRS animation data model and the mathematical operations (Type 0-3 transforms, ZXY rotation order, angle encoding).
*   **Educational Value:** Explains exactly *why* the morph target approach failed ("frame N stores absolute positions for ALL vertices") and why skeletal animation fixes it.
*   **Thorough DoD:** The Definition of Done is comprehensive, catching important cleanup tasks like removing the 302 old `.bin` files that will be orphaned.
*   **Player Path Awareness:** Clearly identifies the need to separate player morph code from boss morph code (Phase 5).

### Weaknesses
*   **Reinventing the Wheel:** Proposes rewriting the GLTF geometry, alpha splitting, and palette texture generation from scratch. Building a binary GLTF writer from the ground up for geometry and materials is highly error-prone and duplicates existing `osrscachereader` logic.
*   **Ad-hoc Coordinate Conversion:** Recommends applying `-z` inversions manually throughout the math, which is prone to missing a sign flip in quaternion or scale calculations.

### Gaps in Risk Analysis & Missing Edge Cases
*   **Missing `animMayaID` check:** Doesn't consider the risk that some sequences might not be frame-based (Maya animations).
*   **GLTF Binary Packing Risk:** Fails to identify the significant risk of writing valid GLTF binary buffers (padding, alignment) from scratch.

### Definition of Done Completeness
*   **Excellent.** Covers all visual, mechanical, performance (>30fps), and cleanup criteria.

## 2. Codex Draft Critique (`SPRINT-022-CODEX-DRAFT.md`)

### Strengths
*   **Pragmatic Export Strategy:** Correctly identifies that we should reuse the existing `GLTFExporter` geometry/material logic and only build a custom `SkeletalBossExporter` for the skinning/animation data. This saves massive amounts of time and reduces risk.
*   **Matrix Math for Basis Conversion:** Uses a clean basis-change matrix (`OSRS_TO_GLTF = new THREE.Matrix4().makeScale(1, -1, -1)`) to handle coordinate system conversion. This is vastly superior and less error-prone than manual sign flipping.
*   **Validation Updates:** Explicitly includes updating `tools/cache-reader/validate-gltf.mjs` to enforce the new skeletal contract, ensuring regressions are caught.
*   **Empty Vertex Groups:** Recognizes the edge case of empty vertex groups and proposes keeping them to maintain stable joint indexing.

### Weaknesses
*   **Less Math Detail:** The explanation of OSRS transforms (Type 0-3) is slightly less explicitly documented than Claude's, glossing over the exact pivot translation math (though it mentions the steps).
*   **File Cleanup:** Misses the cleanup of the legacy `.bin` morph target files in the DoD.

### Gaps in Risk Analysis & Missing Edge Cases
*   **Pivot Application Math:** While the matrix basis conversion is great, the risk of incorrectly applying the rotation/scale pivot (translate to origin -> rotate -> translate back) in Three.js matrix math isn't highlighted as a major risk.

### Definition of Done Completeness
*   **Good, but slightly incomplete.** Misses the cleanup of the 300+ legacy `.bin` files generated by the old morph exporter.

## Conclusion: Which Implementation Strategy is Better?

**Codex (`SPRINT-022-CODEX-DRAFT.md`) has the clearly superior implementation strategy.**

**Why?**
1.  **Code Reuse vs. Rewrite:** Codex's strategy to hook into or fork the existing geometry/material/alpha-split logic from `osrscachereader`'s `GLTFExporter` is the safest and fastest path. Claude's plan to rewrite the entire GLTF binary buffer generation and palette texture extraction from scratch is a massive, unnecessary risk sink.
2.  **Coordinate Math:** Codex's use of a formal change-of-basis matrix (`OSRS_TO_GLTF`) to wrap the raw OSRS transform matrices guarantees mathematical correctness across translations, rotations, and scales. Claude's manual `-z` inversion approach is notoriously buggy when dealing with complex skeletal quaternions.
3.  **Tooling Integration:** Codex explicitly plans to update the `validate-gltf.mjs` script to enforce the new skeletal contract, which is a great CI/CD mindset.

**Recommendation:** Proceed with the **Codex** draft's architecture and implementation plan, but incorporate Claude's more explicit mathematical breakdown of the OSRS angle/rotation encoding and Claude's DoD item for cleaning up the old `.bin` files.
