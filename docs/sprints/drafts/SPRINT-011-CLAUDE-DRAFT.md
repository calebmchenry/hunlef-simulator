# Sprint 011: Fix Hunlef Morph Target Animations & Model Orientation

## Overview

The Corrupted Hunlef GLTF model loads and renders correctly (vertex colors, geometry), but morph target animations don't visually play despite structurally valid animation data. The model also faces the wrong direction. This sprint diagnoses the root cause, fixes the animation pipeline, and corrects the model rotation.

## Root Cause Analysis

### Why morph target animations don't play visually

Investigation of `public/models/corrupted_hunlef.gltf` reveals:

- **302 separate data URI buffers** — Each of the 144 morph targets gets its own `data:application/octet-stream;base64,...` buffer (~27KB POSITION each). This is generated by `osrscachereader`'s GLTFExporter which creates one buffer per attribute. Total: ~5.93 MB across 302 buffers.
- **Morph targets are POSITION-only** — No NORMAL deltas in `targets[].NORMAL`. Three.js will skip `morphNormals` but this is fine for `MeshBasicMaterial`.
- **Animation weight data is correct** — Idle animation: 14 keyframes (0.1s–1.2s), 144 weights per frame, one-hot STEP interpolation. Frame 0 activates target 0 (weight 0.1), frame 1 activates target 15 (weight 1.0). Output accessor count = frames × 144 targets (e.g., 2016 = 14 × 144).
- **Default mesh weights** — `weights[0] = 1`, rest 0. The mesh starts with morph target 0 fully active.
- **Node 0 has no name** — Just `{"mesh": 0}`. The GLTFLoader must resolve animation track bindings by node index.

**Most likely failure points (in order of probability):**

1. **Material replacement destroys morph target program state** — `Renderer3D.loadBossGLTF()` traverses the scene and replaces every mesh's material with a new `MeshBasicMaterial`. In Three.js r183, morph targets are auto-detected from `geometry.morphAttributes`, but the replacement happens *after* the GLTFLoader has already compiled the initial WebGL program. The new material triggers a shader recompile — if `geometry.morphAttributes.position` is empty at that point (because the 302 buffer loads haven't completed), the shader will be compiled *without* morph target support and never recompiled.

2. **302 data URI buffers silently fail or load out-of-order** — Three.js GLTFLoader processes data URIs synchronously via `atob()`, but 302 separate base64 decodes of 27KB each creates significant memory churn. The `bufferView.target = 34962` (ARRAY_BUFFER) on morph target views is correct, but `geometry.morphAttributes.position` may end up as an empty array if any buffer fails to decode.

3. **144 morph targets exceed WebGL uniform/texture limits** — Three.js r150+ uses texture-based morph targets (not uniforms), selecting the top N influential targets per frame. With one-hot encoding, only 1 target is active at a time, so this *should* work. But 144 targets means a very large morph target texture, which may silently fail on some GPUs or hit a Three.js internal limit.

4. **AnimationMixer track binding fails on unnamed node** — The GLTFLoader generates track names like `<nodeName>.morphTargetInfluences[0]`. If node 0 has no name, the track name becomes `.morphTargetInfluences[0]` (empty prefix). The AnimationMixer may fail to resolve this binding against the scene hierarchy.

### Why the model faces the wrong direction

- The GLTF model's Z-axis range is asymmetric: min -513, max 370. The model natively faces **-Z**.
- `Renderer3D.updateBoss()` uses `Math.atan2(dx, dz)` which assumes the model faces **+Z** (Three.js convention).
- Fix: add `Math.PI` offset so `atan2(dx, dz) + Math.PI` rotates the model 180° to face the player.

## Architecture

### Current Pipeline

```
OSRS Cache → osrscachereader GLTFExporter → 302-buffer GLTF (data URIs)
    → Three.js GLTFLoader → Material replacement → AnimationController
```

### Proposed Pipeline

```
OSRS Cache → osrscachereader GLTFExporter → GLTF (302 buffers)
    → Post-export GLB consolidation script → Single-buffer GLB file
    → Three.js GLTFLoader → Material replacement (with morph target preservation)
    → AnimationController (unchanged)
```

**Key change:** A build-time step converts the multi-buffer GLTF into a single-buffer GLB. This eliminates the 302 data URI decode issue, reduces file size (~30% smaller with binary encoding), and is the standard format for GLTF with embedded data.

### Phase 1a: GLB Conversion — Buffer Consolidation (25% effort)

**Goal:** Convert the 302-buffer GLTF into a single-buffer GLB at build time.

**Files:** `tools/cache-reader/export-gltf.mjs`

**Tasks:**
- [ ] After `exporter.export()` produces the GLTF JSON string, parse it
- [ ] Consolidate all 302 data URI buffers into a single `ArrayBuffer`:
  ```javascript
  function consolidateToGLB(gltfJson) {
    const gltf = JSON.parse(gltfJson);
    // Decode all data URI buffers into one contiguous ArrayBuffer
    let totalSize = 0;
    const bufferDatas = gltf.buffers.map(buf => {
      const base64 = buf.uri.split(',')[1];
      const raw = Buffer.from(base64, 'base64');
      totalSize += raw.byteLength;
      return raw;
    });

    // Build consolidated buffer
    const consolidated = Buffer.alloc(totalSize);
    let offset = 0;
    const newOffsets = [];
    for (const data of bufferDatas) {
      newOffsets.push(offset);
      data.copy(consolidated, offset);
      offset += data.byteLength;
    }

    // Update bufferViews to reference single buffer with correct offsets
    gltf.bufferViews.forEach(bv => {
      bv.byteOffset = (bv.byteOffset || 0) + newOffsets[bv.buffer];
      bv.buffer = 0;
    });

    // Replace buffers array with single buffer (no URI — GLB binary chunk)
    gltf.buffers = [{ byteLength: totalSize }];

    // Write GLB binary format
    return writeGLB(gltf, consolidated);
  }
  ```
- [ ] Implement `writeGLB()` to produce the GLB binary container (12-byte header + JSON chunk + BIN chunk, per glTF 2.0 spec)
- [ ] Output as `corrupted_hunlef.glb` instead of `.gltf`
- [ ] Update `Renderer3D.loadBossGLTF()` to load `/models/corrupted_hunlef.glb`
- [ ] Delete the old `.gltf` file
- [ ] Verify: file size should be ~4 MB (vs ~8 MB with base64 overhead)

## Open Questions

1. **Is the issue actually in the GLTFLoader or the material replacement?** Phase 0 diagnostics will answer this definitively. If `geometry.morphAttributes.position` is populated after load but before material swap, the issue is in the material. If it's empty from the start, the issue is in the loader/data.

2. **Should we validate the GLB with an external tool?** Running the GLTF through `gltf-validator` or loading it in Don McCurdy's glTF Viewer would confirm whether the animation data works outside of our codebase. This isolates Three.js issues from data issues.

3. **Can we reduce the 144 morph targets?** The animations use one-hot encoding, so each keyframe activates exactly one target. If multiple animations reuse the same targets, the total unique targets needed may be much less than 144. However, the OSRS cache defines them, so deduplication would require careful analysis.

4. **Should the GLB conversion be a build step or a one-time script?** If we don't expect to re-export from the OSRS cache frequently, a one-time conversion is simpler. If the export script is run regularly, integrating GLB conversion into it makes more sense. Recommend: integrate into `export-gltf.mjs` so any re-export automatically produces GLB.

5. **Does the `morphTargetsRelative` flag need to be explicitly set?** GLTF morph targets are always relative (deltas), and Three.js GLTFLoader should set `geometry.morphTargetsRelative = true`. Phase 0 diagnostics will confirm this.
